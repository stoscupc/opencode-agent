### Join as review

```bash
cd devtools && bin/gantry session join review --focus "commit review and code quality"
```

### Phase 1: Commit Pending Work

Scan for uncommitted changes and produce clean, small, semantically clear commits.

#### Check for pending changes

```bash
git status -s
git diff --stat
```

#### Commit guidelines

- **Conventional commit format**: `type(scope): short description`
- **Types**: `feat`, `fix`, `test`, `docs`, `refactor`, `chore`
- **Scope**: project name (`hmc-sim`, `z-tf-expert`, `iocp-sim`, `devtools`, `demo-app`, `z-memory`, etc.)
- **No AI branding**: Do NOT include `Co-Authored-By`, `Generated by`, or any tool attribution
- **Small chunks**: One logical change per commit. Split large diffs by project or concern.
- **Subject line**: Under 72 characters, imperative mood, lowercase after type prefix
- **Stage specific files**: Use `git add <file>...`, never `git add -A` or `git add .`
- **Fix before commit**: Run lint/fmt and fix issues before committing. Do not use `--no-verify`.
- **On hook failure**: Fix the issue, re-stage, create a NEW commit (never amend the previous one)

#### Commit workflow per chunk

```bash
# 1. Identify a logical group of changes
git diff --stat -- <project>/

# 2. Stage specific files
git add <file1> <file2> ...

# 3. Commit with conventional format (use heredoc for message)
git commit -m "$(cat <<'EOF'
type(scope): concise description
EOF
)"

# 4. If pre-commit hook fails, fix and retry as NEW commit
```

#### Splitting strategy

- One commit per project when changes are independent
- Separate `feat` from `test` from `docs` within the same project
- Group tightly coupled cross-project changes (e.g., API + client) into one commit

### Phase 2: Code Review Recent Commits

Review the last 5 commits on the current branch for quality issues.

#### Inspect recent commits

```bash
git log --oneline -5
git show --stat HEAD~4..HEAD
```

#### For each commit, review

```bash
git show <sha> --stat   # what files changed
git show <sha>          # full diff
```

#### Review checklist per commit

1. **Commit message quality**: Follows conventional format? Semantically clear? No branding?
2. **Scope correctness**: Does the scope match the files changed?
3. **Atomic change**: Is this one logical change, or should it have been split?
4. **Code quality**: Any obvious bugs, security issues, missing error handling?
5. **Test coverage**: Did a `feat` or `fix` come with tests? Should it have?
6. **Lint cleanliness**: Any lint suppressions or `_ =` ignores that hide real issues?
7. **Dead code**: Any commented-out code, unused imports, TODO placeholders left behind?
8. **Naming**: Are new symbols (functions, types, variables) clearly named?

#### Create review workflow and tasks

Organize all review findings into a workflow with workstream-grouped tasks.

##### 1. Create a review workflow

Before creating any tasks, create a single workflow for this review session via `workflow_create`:

- **title**: `Code review: <commit-range>` (e.g., `Code review: abc1234..def5678`)
- **workstream_id**: `code-review`
- **tags**: `["review"]`

##### 2. Group issues by workstream

Map each issue to a workstream based on the project directory it affects:

| Directory | Workstream ID |
|-----------|--------------|
| `hmc-sim/` | `hmc-sim` |
| `z-tf-expert/` | `z-tf-expert` |
| `devtools/` | `devtools` |
| `z-pkg/` | `z-pkg` |
| `web/` | `web` |
| `iocp-sim/` | `iocp-sim` |
| `z-memory/` | `z-memory` |
| `z-manuals-kb/` | `z-manuals-kb` |
| `gantry/` | `gantry` |
| Cross-project | `repo` |

> **Parallel codebases — gantry + devtools:** The `gantry/` and `devtools/` directories
> contain mirrored code (see the "Parallel Codebase" section in each project's CLAUDE.md).
> When a commit touches files in both, create a **single** task titled
> `[review] gantry+devtools: <description>` with workstream `devtools` covering both
> codebases. Do NOT create separate duplicate tasks for the same logical change in each
> directory.

##### 3. Create tasks under the workflow

Use `task_create` with `workflow_id` and `workstream_id` for each issue:

- **title**: `[review] <project>: <concise issue description>`
- **workflow_id**: The review workflow ID from step 1
- **workstream_id**: The project workstream from the table above
- **role**: `bugfix` for bugs/security, `impl` for missing tests/features, `test` for test gaps, `refactor` for code quality
- **priority**: 1 for security/correctness, 2 for missing tests, 3 for style/quality
- **description**: What's wrong, which commit introduced it, suggested fix
- **files**: The affected file paths
- **tags**: `["review-followup"]` (plus any additional tags)

##### 4. Complete the review workflow

After all tasks are created:
1. Broadcast a `lint` result summarizing the review
2. Mark the review workflow as completed via `workflow_complete`

#### Broadcast review summary

```bash
# MCP: broadcast_result kind="lint" with review outcome
```

### Interactive Workflow

1. Join as review
2. **Commit phase**: Scan `git status`, commit pending changes in small conventional chunks
3. **Review phase**: Read last 5 commits, review each against checklist
4. Create a review workflow via `workflow_create`
5. Create tasks grouped by workstream under the review workflow
6. Broadcast summary of commits made and issues found
7. Complete the review workflow

## Autonomous Agent Protocol

When running as a spawned autonomous agent, execute these steps in order:

1. **Join session**: Use `session_join` with role `review` and focus set to "commit review and code quality".
2. **Claim task**: If a review task is assigned, use `task_claim`. If no tasks, proceed with default workflow (commit + review last 5).
3. **Commit phase**:
   a. Run `git status -s` and `git diff --stat` to find uncommitted work.
   b. Group changes into small logical chunks by project/concern.
   c. For each chunk: stage specific files, commit with conventional format, no branding.
   d. Fix any pre-commit hook failures before retrying.
   e. Broadcast a `build` result for each project committed.
4. **Review phase**:
   a. Run `git log --oneline -5` to get recent commits.
   b. For each commit: `git show <sha>` and review against the checklist.
   c. Create a review workflow via `workflow_create`:
      - **title**: `Code review: <first-sha>..<last-sha>`
      - **workstream_id**: `code-review`
      - **tags**: `["review"]`
   d. For each issue found: create a task via `task_create` with:
      - **workflow_id**: the review workflow ID from step (c)
      - **workstream_id**: the project slug (e.g., `devtools`, `hmc-sim`, `web`, `z-pkg`)
      - **tags**: `["review-followup"]`
      - Appropriate role and priority per the review checklist
   e. **Track recommendations**: When approving with non-blocking recommendations, create a follow-up task for EACH recommendation.
   f. Broadcast a `lint` result summarizing the review.
   g. Complete the review workflow via `workflow_complete`.
5. **Complete task**: Use `task_complete` with summary of commits made and issues found.
6. **Leave session**: Use `session_leave` before exiting.
